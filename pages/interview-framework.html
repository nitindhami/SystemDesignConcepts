System Design Interview Framework

  1. Clarify scope: confirm core users, must-have features, and constraints.
  2. Estimate: request volume, storage growth, read/write split, latency goals.
  3. Sketch architecture: API, services, databases, caches, queues.
  4. Deep dive: data model, bottlenecks, failure handling, consistency.
  5. Tradeoffs: explain alternatives and why you made each choice.
  6. Close strong: summarize design, risks, and next optimization steps.

Pattern Chooser (Quick Interview Guide)

  • Strategy: interchangeable business rules (pricing, ranking).
  • Observer: one event updates multiple subscribers.
  • Factory: object creation hidden behind interfaces.
  • Command: queueing/retrying actions with auditability.
  • Proxy: add caching, access control, or lazy loading in front of core object.

Common Mistakes in Interviews

  • Jumping to components before requirements are clear.
  • Choosing tools by brand name instead of workload patterns.
  • Ignoring non-functional goals like reliability and operability.
  • Not discussing failure modes, retries, and observability.
  • Over-detailing one area while neglecting end-to-end flow.

How to Think Like an Interviewer

Interviewers look for structure, pragmatic decisions, clear assumptions, and ability to adapt when constraints change.

Cheat sheet: communicate tradeoffs and failure handling, not just happy-path architecture.
